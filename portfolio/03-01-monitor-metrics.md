# Azure Monitor：メトリック監視（VM / Storage / Network）

## 1. 目的
Azure リソースの状態を **数値（メトリック）で継続的に把握できるようにする**。  
アラート設定の前段階として、どの値を見れば「正常／異常」を判断できるかを整理する。

---

## 2. 設計

### 2-1. 対象リソース
- **VM（仮想マシン）**
  - CPU 使用率
  - メモリ使用量（ゲストレベル）
  - ディスク IOPS / 待ち時間
- **Storage アカウント（Blob）**
  - 読み取り／書き込み要求数
  - スループット（トランザクション数）
  - エラー数（失敗した要求）
- **Virtual Network / NIC**
  - 受信／送信バイト数
  - パケット数
  - ドロップされたパケット数

### 2-2. 監視の粒度
- **集計間隔**：1 分 / 5 分 / 15 分  
- **表示期間**：直近 1 時間 / 24 時間 / 7 日  
- **表示方法**：折れ線グラフ（時系列）＋複数メトリックの重ね合わせ

---

## 3. 前提条件（Prerequisites）

- Azure サブスクリプションに以下が存在していること：
  - 監視対象の **VM / Storage / VNet / NIC**
- Azure Monitor が有効（デフォルトでメトリックは収集される）
- ロール：
  - **閲覧者（Reader）以上**の権限があること

---

## 4. 手順（How：GUI）

### 4-1. Azure Monitor のメトリック画面を開く

1. Azure ポータルへサインイン  
2. 左メニューから **「Monitor（監視）」** を選択  
3. 上部タブから **「メトリック（Metrics）」** をクリック  

📌 **画像を入れる箇所**  
`../images/02-01-monitor-metrics-01-open.png`

---

### 4-2. 対象リソースの選択

1. メトリック画面上部の **「スコープ」欄** に、  
   **サブスクリプション → リソース グループ → リソース** の階層が表示されている。
2. 階層の最下層にある**リソースタイプ（仮想マシン、ストレージ アカウント、ネットワーク インターフェイス）**から、監視したいリソースを選択する。
3. **「適用」ボタン** をクリックして確定。

📌 **画像を入れる箇所**  
`../images/02-01-monitor-metrics-02-scope.png`

---

### 4-3. メトリックの選択

1. 「メトリック」ドロップダウンを開く  
2. VM の場合：
   - **CPU 使用率（Percentage CPU）**
   - **ディスクの読み取り／書き込みバイト数**
3. Storage の場合：
   - **Total requests**
   - **Success E2E Latency / Server Latency**
4. NIC の場合：
   - **Bytes sent / Bytes received**
   - **Packets dropped**

📌 **画像を入れる箇所**  
`../images/02-01-monitor-metrics-03-metric-select.png`

---

### 4-4. 表示期間・集計間隔の設定

1. 画面右上の期間選択で **「過去 24 時間」などを選択**  
2. 「時間の粒度」で **5 分** を選ぶ  
3. グラフが自動更新され、時系列でメトリックが表示される  

📌 **画像を入れる箇所**  
`../images/02-01-monitor-metrics-04-time-range.png`

---

### 4-5. 複数メトリックの重ね合わせ

1. 「メトリックの追加（Add metric）」ボタンをクリック  
2. 同じリソースに対して別のメトリックを選択  
   - 例：VM で **CPU 使用率** と **ディスク待ち時間** を同時表示  
3. グラフ上で色分けされて表示される  

📌 **画像を入れる箇所**  
`../images/02-01-monitor-metrics-05-multi-metric.png`

## 5. メトリック監視で分かること（So What）

### 5-1. VM のボトルネック把握
- **CPU 使用率が常に高い** → スケールアップ／スケールアウト検討  
- **ディスク待ち時間が高い** → ストレージ性能不足、ディスク種別の見直し  
- **ネットワーク送受信が急増** → 負荷テスト／攻撃／大量データ転送の可能性

### 5-2. Storage の性能・安定性
- **要求数が急増** → アプリケーションの負荷増加  
- **レイテンシが高い** → 設計見直し（アクセスパターン、冗長化）  
- **エラー数が増加** → アプリ側のリトライ制御や例外処理の確認

### 5-3. ネットワークの異常兆候
- **パケットドロップ増加** → ネットワーク帯域不足、NSG／Firewall 設定の影響  
- **送受信バイト数の急増** → 想定外の通信（ログ調査のトリガー）

---

## 6. 気づき・学び（Learning）

- メトリック監視は **「今の状態を知る」ための最初の入口**。  
- どの値が「正常」で、どこから「異常」なのかを自分で定義することで、  
  後続の **アラートルール設計** や **キャパシティ計画** に繋げられる。  
- 単一メトリックではなく、**複数メトリックの組み合わせ**で原因を推定する視点が重要。
